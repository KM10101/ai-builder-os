# Claude Code 常用命令速查

> 这页优先服务于“快速上手 + 实战速查”，不追求覆盖全部能力，而是先覆盖高频有用内容。

## Claude Code 更适合做什么

默认更适合优先承担：

- 读代码
- 梳理调用链
- 拆任务
- 给改动方案
- 做代码 review
- 帮你识别风险点

## 高频 / 命令

Claude Code 提供内置 / 命令，可以在会话中直接使用：

### /init

```bash
/init
```

**用途**：快速为当前项目生成 `CLAUDE.md` 文件

**适用场景**：
- 刚 clone 一个项目，想要快速建立项目上下文
- 项目没有 CLAUDE.md，需要初始化
- 想快速让 Claude Code 理解项目结构

**补充说明**：会扫描项目并生成包含项目结构、技术栈、关键文件等信息的 CLAUDE.md

---

### /review

```bash
/review
```

**用途**：对当前 git 改动进行代码 review

**适用场景**：
- 提交前想先自查
- 看看哪些改动还没 commit
- PR 之前做一轮预审

**补充说明**：会分析 uncommitted 的改动并给出 review 意见

---

### /test

```bash
/test
```

**用途**：为当前改动补测试或运行现有测试

**适用场景**：
- 改完后需要补测试
- 想运行相关测试验证改动
- 不知道测试应该怎么写

**补充说明**：会根据改动内容推荐测试方案

---

### /commit

```bash
/commit
```

**用途**：帮助生成 git commit 信息并提交

**适用场景**：
- 改动完成后需要提交
- 不知道 commit 信息怎么写
- 想快速完成 commit

**补充说明**：会先展示改动摘要，然后帮你写 commit message

---

### /undo

```bash
/undo
```

**用途**：撤销最近一次的 AI 改动

**适用场景**：
- 刚才的修改不满意
- 想回退到修改前的状态
- 不小心让 AI 改了不想改的内容

**补充说明**：会撤销最近的 Edit 操作

---

### /lint

```bash
/lint
```

**用途**：运行项目的 lint 检查

**适用场景**：
- 提交前想跑 lint
- 想知道代码有没有格式问题
- 想让 AI 帮忙修 lint 错误

**补充说明**：会根据项目配置运行对应 lint 工具

---

### /context

```bash
/context
```

**用途**：查看当前会话的上下文使用情况

**适用场景**：
- 想了解已经用了多少 context
- 需要清理上下文
- 想评估是否应该开新会话

---

### /clear

```bash
/clear
```

**用途**：清空当前会话历史

**适用场景**：
- 想开始一个新话题
- 上下文太长了想重置
- 想让 AI 重新理解需求

**补充说明**：会清除当前会话的所有消息，但保持项目上下文

---

### /resume

```bash
/resume [session-id]
```

**用途**：恢复之前的会话

**适用场景**：
- 想继续之前的任务
- 需要查看之前对话
- 想复用之前的上下文

---

### /git

```bash
/git <command>
```

**用途**：直接在会话中执行 git 命令

**适用场景**：
- 想查 git 历史
- 想看某个 commit 的改动
- 想做 git 操作但不想切出窗口

**补充说明**：相当于在终端执行 git 命令，但保持在 Claude Code 会话中

## 高频使用方式

### 1. 在项目目录启动

```bash
claude
```

建议进入项目后第一句话不要直接让它改代码，而是先做理解。

### 2. 读代码模式

典型起手：

```text
先不要修改代码，先帮我理解这个仓库里和 XXX 相关的实现。
```

适合：
- 接手陌生模块
- 找入口
- 看调用链
- 理解现状

### 3. 方案模式

典型起手：

```text
我想实现 XXX。先不要直接写代码，请先告诉我需要修改哪些文件、每个文件要改什么、风险点是什么。
```

适合：
- 小需求改动前先收敛范围
- 改动已有逻辑前先看影响

### 4. Review 模式

典型起手：

```text
请 review 这次改动，优先指出高风险问题、边界条件遗漏和潜在兼容性问题。
```

适合：
- 自己改完后复查
- 看别人的改动
- PR review 前预审

## 推荐高频 prompt 片段

### 先理解，不动代码

```text
先不要改代码。
```

### 限定范围

```text
只看和当前问题直接相关的模块，不要扩展到无关部分。
```

### 输出关键文件

```text
请标出最关键的 3 个文件，并解释各自作用。
```

### 要求风险分析

```text
请指出你认为最容易出问题的点。
```

### 要求先出计划

```text
先输出修改计划，等我确认后再执行。
```

## 常见场景快速起手

| 场景 | 推荐起手方式 |
|------|-------------|
| 刚接手新项目 | `/init` 或 `先不要改代码，帮我理解这个项目的结构和主要模块` |
| 改需求前先了解影响 | `先不要改代码，告诉我如果要改 XXX，需要注意什么，风险点有哪些` |
| 提交前自检 | `/review` 或 `请 review 这次改动，优先指出高风险问题` |
| 补测试 | `/test` 或 `请帮我补一下相关测试` |
| 快速提交 | `/commit` |
| 发现改错了 | `/undo` |
| lint 有问题 | `/lint` |
| 上下文太乱想重新开始 | `/clear` |

## 实战建议

### 适合 Claude Code 的默认节奏

1. 先理解现状
2. 再输出方案
3. 最后 review 执行结果

### 常见坑

- 一上来就让它直接实现
- 没有限定分析范围
- 没要求它指出风险点
- 对大任务没有先拆小

## 一个推荐定位

把 Claude Code 当成：

- **代码理解助手**
- **任务拆解助手**
- **高级 review 搭子**

而不是默认把它当“全自动写代码工具”。

## 示例用法

### 示例 1：阅读陌生接口

```text
先不要修改代码。帮我阅读和订单超时取消相关的实现：
1. 找入口文件和关键调用链
2. 解释当前处理逻辑
3. 标出最关键的 3 个文件
4. 说出最容易出问题的点
```

### 示例 2：改动前先做方案

```text
我想给这个接口增加一个状态字段。先不要直接写代码，请先输出：
1. 要修改哪些文件
2. 每个文件大概要改什么
3. 有没有兼容性风险
4. 测试应该怎么补
```

### 示例 3：做一次 Review

```text
请 review 这次改动，重点看：
- 是否有逻辑漏洞
- 是否遗漏边界条件
- 是否有兼容性风险
- 是否需要补测试
请优先给高风险问题。
```
