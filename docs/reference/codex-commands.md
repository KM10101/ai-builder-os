# Codex 常用命令速查

> 这页优先记录面向真实开发的高频用法，目标是让你在做任务时能快速想起"该怎么和 Codex 协作"。

## 命令行基本用法

### 直接执行

```bash
codex "你的任务描述"
```

适合快速执行一个明确任务。

### 非交互模式 exec

```bash
codex exec [OPTIONS] "任务描述"
```

适合 CI/CD、脚本等非交互场景。

常用参数：
- `-m, --model <MODEL>` - 指定模型
- `-s, --sandbox <MODE>` - 沙箱模式 (read-only / workspace-write / danger-full-access)
- `-a, --ask-for-approval <POLICY>` - 审批策略 (never / on-request / on-failure)
- `--full-auto` - 全自动模式，等同于 `-a on-request -s workspace-write`
- `-C, --cd <DIR>` - 指定工作目录
- `--json` - JSON 格式输出
- `-o, --output-last-message <FILE>` - 把最后输出写入文件

### 代码 Review

```bash
codex review [OPTIONS] "review 指令"
```

常用参数：
- `--uncommitted` - 审查所有未提交的改动
- `--base <BRANCH>` - 对比指定分支
- `--commit <SHA>` - 审查指定 commit
- `--title <TITLE>` - review 报告标题

### 会话管理

```bash
# 继续上一个会话
codex exec --resume --last

# 恢复指定会话
codex exec --resume <session-id>

# Fork 一个会话
codex fork <session-id>
```

### MCP 服务

```bash
# 列出已配置的 MCP 服务器
codex mcp list

# 添加 MCP 服务器
codex mcp add <name> <command>

# 移除 MCP 服务器
codex mcp remove <name>
```

## Codex 更适合做什么

默认更适合优先承担：

- 执行明确改动
- 实现小功能
- 修明确 bug
- 补测试
- 按方案推进落地

## 高频使用方式

### 1. 在项目目录启动 / 使用

先在目标仓库目录下工作，然后再让 Codex处理明确任务。

### 2. 执行模式

典型起手：

```text
按下面要求修改代码，尽量少改动，保持现有风格，不要顺手做无关重构。
```

适合：
- 明确小改动
- 已经知道要改哪些地方
- 已有一版方案，准备执行

### 3. 先计划再执行

典型起手：

```text
先列出你准备修改的文件和原因，等我确认后再执行。
```

适合：
- 不想让它直接开改
- 想先看边界和范围
- 想降低改偏概率

### 4. 执行后自查

典型起手：

```text
改完后请总结修改点、风险点，以及建议我执行哪些验证命令。
```

适合：
- 任何代码修改任务之后
- 尤其适合小需求和 bug 修复

## 推荐高频 prompt 片段

### 限制改动范围

```text
只修改和本次需求直接相关的文件,不要扩展到无关模块。
```

### 禁止顺手重构

```text
不要顺手做无关优化或重构。
```

### 保持代码风格

```text
保持现有代码风格和项目约定。
```

### 要求总结改动

```text
请用简洁的方式总结本次改动点。
```

### 要求我重点 review 的地方

```text
请列出需要我重点 review 的地方。
```

## 实战建议

### 更稳的默认节奏

1. 目标具体
2. 边界明确
3. 先计划再执行
4. 改完后自查

### 常见坑

- 需求描述过大
- 没说清楚哪些地方不能动
- 没明确要求不要顺手重构
- 改完后没让它总结和自查

## 一个推荐定位

把 Codex 当成:

- **高执行力工程助手**
- **明确任务落地器**
- **测试补全和小步迭代帮手**

## 示例用法

### 示例 1:执行一个小改动

```text
按下面要求修改代码:
- 给订单接口增加一个 `source` 字段透传
- 只修改和这个需求直接相关的文件
- 保持现有代码风格
- 不要顺手做无关重构
- 改完后总结修改点和风险点
```

### 示例 2:先计划再执行

```text
我需要修复这个 bug。先列出你准备修改的文件、原因和可能风险,等我确认后再执行。
```

### 示例 3:执行后自查

```text
改完后请告诉我:
1. 本次改了哪些文件
2. 每个改动的作用
3. 需要我重点 review 的地方
4. 建议执行哪些验证命令
```
