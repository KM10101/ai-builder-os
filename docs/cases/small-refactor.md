## 示例填法（参考）

### 背景
某个 service 方法过长、重复逻辑较多，虽然暂时能跑，但后续需求继续叠加会越来越难维护。

### 工具分工
- 先让 Claude Code 评估是否值得重构，并提出小范围方案
- 再让 Codex 执行明确的收敛式改动
- 最后让 Claude Code review

### 使用的 Prompt
先不要直接改代码。请帮我判断这段实现是否值得做一次小范围重构，并给出一个尽量低风险的改动方案。

### 有效点
- “先判断是否值得重构”这个步骤很重要
- 限定“小范围重构”比直接说“帮我重构”稳定很多

### 下次优化
下次可以更早限制“只允许提炼重复逻辑，不调整对外行为”。
