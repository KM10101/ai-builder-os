# AI Coding Preflight Checklist（草案）

这份 checklist 用于在开始一次 AI coding 实验或实际编码前，先做一轮**轻量前置检查**。

目的不是增加流程负担，而是尽量在编码开始前就挡掉一批低级问题，避免把环境问题、工具问题或实验对象问题误当成 prompt / workflow 问题。

它和 [`ai-coding-rules.md`](./ai-coding-rules.md) 以及 [`ai-coding-review-checklist.md`](./ai-coding-review-checklist.md) 分工不同：

- **preflight checklist**：开始前先确认基础条件是否成立
- **rules**：约束 AI 编码时的默认行为
- **review checklist**：代码生成后做快速人工验收

---

## 使用时机

适用于以下场景：

- 开始一个新的 AI coding 小实验前
- 准备让 AI 对已有项目做一次明显修改前
- 更换开发环境、编辑器、模型或 AI coding 工具后
- 发现最近实验结果不稳定，想先排查是不是基础条件有问题

建议在每次正式开始前，花 3 到 5 分钟快速过一遍。

---

## 一、任务与目标是否清楚

- [ ] 这次任务的目标是否能用 1 到 3 句话说清楚
- [ ] 当前范围是否明确
- [ ] 当前不做的内容是否明确
- [ ] 是否知道这次更看重“功能跑通”还是“流程验证 / 质量验证”

### 重点问题

- 这次到底要验证什么？
- 是在做功能迭代，还是在做 workflow 实验？
- 如果结果不理想，应该归因为功能问题，还是流程问题？

---

## 二、开发环境是否稳定可用

- [ ] 当前开发环境是否已经可以正常进入和使用
- [ ] 基础工具是否可用（如 `git`、`node`、`npm` / `pnpm`、Python 等）
- [ ] 项目目录是否能正常打开、编辑和运行
- [ ] 依赖安装方式是否明确
- [ ] 如果使用的是新环境（如 WSL），是否已经验证过最小运行链路

### 重点问题

- 今天的实验会不会先被环境问题打断？
- 当前环境是否已经足够稳定，不会让结果掺入太多噪音？

---

## 三、AI coding 工具链是否就绪

- [ ] 使用哪一个 AI coding 工具已经明确（如 Codex、Claude Code、Zed 等）
- [ ] 工具当前能否正常连接目标工作目录
- [ ] 是否知道这次工具运行在本地、WSL、远程 Linux 还是其他环境
- [ ] 如工具近期有断连 / 重连问题，是否已有临时规避方案
- [ ] 是否清楚当前工具的权限模式和行为边界

### 重点问题

- 工具本身今天是否处于“能稳定干活”的状态？
- 如果中断，是偶发问题，还是环境配置还没收口？

---

## 四、权限策略是否提前确认

- [ ] 是否清楚这次编码是否需要较高权限
- [ ] 是否已经决定采用哪种权限策略（严格确认 / 一次性放开 / 受控放开）
- [ ] 是否接受在长任务中频繁人工确认带来的打断成本
- [ ] 是否明确哪些操作必须保留人工确认

### 重点问题

- 这次是要“低风险探索”，还是要“提高连续生成效率”？
- 权限策略如果不提前想清楚，会不会让实验过程被大量确认弹窗打断？

---

## 五、实验对象是否适合当前方案

- [ ] 本次处理对象是否适合当前技术路线
- [ ] 如果涉及网页抓取，是否已初步判断页面是否依赖 JS 渲染
- [ ] 如果涉及外部接口，是否知道认证、频率限制和返回稳定性
- [ ] 如果涉及测试样本，是否已经准备好最小样本集
- [ ] 是否存在容易误导实验判断的特殊样本

### 重点问题

- 失败到底是实现问题，还是目标对象本身不适合当前方案？
- 当前样本是否足够代表你想解决的真实问题？

---

## 六、运行与验证方式是否提前想清楚

- [ ] 是否知道完成后要怎么验证主链路
- [ ] 是否已经准备好至少一个最小输入样例
- [ ] 是否明确“成功”长什么样
- [ ] 是否明确这次最少要留下什么产出（代码、文档、记录、结论）

### 重点问题

- 如果 AI 交付了一堆代码，我要怎么快速判断它到底成没成？
- 这次验证是看“能跑”，还是看“质量是否可接受”？

---

## 七、记录与复盘入口是否准备好

- [ ] 今日文档是否已经建好
- [ ] 是否知道今天要记录哪些关键观察
- [ ] 是否明确今天结束时至少要留下一个有效结论
- [ ] 如果今天偏离原计划，是否准备记录偏差原因

### 重点问题

- 今天如果没按计划推进，最后会不会什么也没沉淀下来？
- 这次实验失败时，能不能留下对下一次有价值的记录？

---

## 最小通过标准

如果下面几项都能回答“是”，通常就可以开始：

- [ ] 任务目标清楚
- [ ] 开发环境可用
- [ ] AI coding 工具可用
- [ ] 权限策略明确
- [ ] 已知如何验证结果

如果其中有 2 项以上回答不了，建议先不要急着开始写代码，先补基础条件。

---

## 常见需要先暂停的问题

以下情况建议先停一下，别直接开干：

- 还说不清这次到底在验证什么
- 开发环境刚搭好，但还没验证最小运行链路
- AI 工具连接不稳定，随时可能断
- 权限策略没有想清楚，预计会被频繁确认打断
- 目标网页 / 接口本身就不适合当前方案
- 做完之后也不知道该怎么验收

---

## 当前版本的定位

这是一份偏轻量的 preflight checklist，适合：

- 最小实验项目
- 小型工具 / CLI / LLM 应用 MVP
- AI coding workflow 的早期摸索阶段

后续可以根据实践继续收敛：

- 真正有用的项留下
- 重复、空泛、无助于判断的项删掉
- 按不同实验类型拆成更具体的版本
