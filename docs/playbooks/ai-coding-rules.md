# AI Coding Rules（草案）

这份规则文件用于约束 AI coding tool 在生成项目代码时的默认交付行为。

当前目标不是把规则写得很重，而是先形成一套**最小有效约束**，让输出质量、安全性和可维护性有基本保障。

## 使用方式

在开始让 AI 编写项目代码前：

1. 先提供任务文档（目标、范围、限制、交付标准）
2. 再明确要求 AI 遵守本规则文件
3. 要求 AI 在编码前先输出实现计划
4. 完成后按本规则进行自检

## 最小规则集

### 1. 先计划，后编码
- 先输出实现计划，再开始写代码
- 说明拟修改或新增的文件
- 说明模块划分和每个模块职责
- 说明主要风险点

### 2. 只实现当前范围
- 只实现任务文档中明确要求的内容
- 不擅自扩展未来功能
- 不因为“看起来顺手”就增加复杂设计

### 3. 输入必须校验
- 对 URL、路径、命令行参数、环境变量等外部输入做基本校验
- 对非法输入给出明确错误信息

### 4. 外部调用必须可控
- 网络请求、文件读写、模型调用必须有错误处理
- 网络请求必须设置超时
- 不写危险或不可解释的 shell 调用

### 5. 配置与密钥不得硬编码
- API key、base URL、模型名等从环境变量或配置文件读取
- 不把敏感信息写进源码或测试样例

### 6. 输出要模块化且可维护
- 避免把所有逻辑塞进单文件
- 函数职责保持单一
- 命名清晰，避免过度抽象

### 7. 必须提供最小验证
- 至少提供一条主链路验证方式
- 如适合，补最小测试或验证脚本
- 说明如何运行验证

### 8. 必须补齐交付说明
- 更新 README 或等价运行说明
- 写清楚如何配置、运行、验证
- 写清楚已知限制和暂不支持范围

### 9. 提交前自检
交付前必须输出：
- 做了什么
- 没做什么
- 已知限制
- 风险点
- 如何验证

## 人工审核重点

人工不需要通读全部代码，重点只看：

1. 主链路是否符合原始目标
2. 模块边界是否清楚
3. 风险点是否有明确处理
4. 自动验证是否通过
5. 已知限制是否写清楚

## 当前适用场景

适用于：
- 最小实验项目
- CLI 工具
- 小型服务原型
- LLM 应用 MVP

暂不针对：
- 大型复杂系统
- 高并发生产服务
- 强合规 / 强安全审计场景

后续可以根据实践继续收敛或补充。
