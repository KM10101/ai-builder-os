# AI Coding Review Checklist（草案）

这份 checklist 用于在 AI 生成代码后做**快速人工验收**。

目标不是人工通读全部代码，而是用尽量少的精力，检查最关键的风险点和交付完整度。

## 使用原则

- 优先检查主链路，而不是逐行阅读实现细节
- 优先检查高风险位置，而不是平均分配注意力
- 优先看自动验证结果，而不是完全依赖肉眼判断
- 如果核心项不通过，直接退回返工，不进入“凑合可用”状态

## 一、目标与范围检查

- [ ] 当前实现是否符合原始任务目标
- [ ] 是否只实现了当前范围，没有擅自扩展
- [ ] 是否明确写出了不支持 / 暂不处理的范围

### 重点问题
- 这是不是我要的东西？
- 有没有做偏？
- 有没有“顺手加了很多未来功能”？

## 二、主链路检查

- [ ] 从输入到输出的主流程是否跑通
- [ ] 主链路是否与任务文档描述一致
- [ ] 样例输入是否能得到预期类型的结果
- [ ] 关键输出格式是否稳定

### 重点问题
- 它真的能完成这次任务吗？
- 结果是不是“形式上能跑”，但实际偏题？

## 三、结构与可维护性检查

- [ ] 模块划分是否清楚
- [ ] 是否避免把大量逻辑塞进单一文件
- [ ] 函数和文件命名是否清晰
- [ ] 是否存在明显过度抽象或奇怪耦合

### 重点问题
- 后续我要改它的时候，会不会很痛苦？
- 是否已经有“demo 代码味道过重”的迹象？

## 四、风险与安全检查

- [ ] 外部输入是否有基本校验
- [ ] 网络请求 / 模型调用是否设置超时
- [ ] 外部调用是否有错误处理
- [ ] 是否存在硬编码密钥、token、URL 或敏感信息
- [ ] 是否存在危险 shell 拼接或不可解释操作

### 重点问题
- 有没有明显不安全的写法？
- 有没有后面很容易出事故的地方？

## 五、验证与交付完整度检查

- [ ] 是否提供最小验证方式（测试、脚本或样例运行）
- [ ] README / 运行说明是否更新
- [ ] 配置方式是否清楚
- [ ] 已知限制是否写明
- [ ] AI 是否输出了自检结果和风险说明

### 重点问题
- 我能不能不猜就把它跑起来？
- 后续的我能不能快速理解这次交付？

## 六、是否接收这次交付

满足下面条件再接收：

- [ ] 主链路正确
- [ ] 没有明显安全问题
- [ ] 自动验证已通过或已有合理解释
- [ ] 文档基本完整
- [ ] 已知限制明确

如果以上关键项有任何一项不满足，优先返工，而不是先合并再说。

## 适合返工的常见情况

以下情况建议直接要求 AI 返工：

- 主链路和原始目标不一致
- 输出虽然能跑，但结构混乱、难以维护
- 缺少超时 / 错误处理 / 输入校验
- README 没更新，交付信息不完整
- 没有任何验证方式
- 已知限制没有写，导致边界模糊

## 备注

这份 checklist 的目的不是追求完美，而是帮助你用较低成本挡掉一批常见问题。

后续可以根据实践，把真正有效的检查项留下，把没用或过重的项删掉。
