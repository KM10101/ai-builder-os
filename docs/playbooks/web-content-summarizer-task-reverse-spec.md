# Web Content Summarizer：反推任务文档（草案）

这份文档不是项目介绍，而是站在 **“重新把任务交给 AI coding tool”** 的角度，反推出来的一版任务说明。

目标是让 AI 在开始编码前，就清楚知道：
- 要做什么
- 不要做什么
- 技术边界是什么
- 交付标准是什么
- 需要满足哪些质量与安全约束

建议与以下文档配套使用：

- [AI Coding Rules（草案）](./ai-coding-rules)
- [AI Coding Review Checklist（草案）](./ai-coding-review-checklist)

---

## 1. 任务目标

请实现一个最小可运行的网页内容总结工具。

用户输入一个网页 URL，程序需要完成以下流程：

1. 获取网页内容
2. 提取网页标题与主要正文
3. 调用 LLM 生成结构化总结
4. 输出结构化结果，并在需要时保存运行产物

本任务的重点不是做完整产品，而是完成一条 **可运行、可验证、可复盘** 的最小链路。

---

## 2. 功能范围

### 本次必须实现

1. **命令行运行入口**
   - 用户可以通过 CLI 输入一个 URL
   - 支持最小必要参数

2. **网页内容抓取**
   - 请求并获取目标网页 HTML
   - 支持基础错误处理
   - 支持请求超时

3. **标题与正文提取**
   - 尽量提取网页标题
   - 尽量提取适合总结的主要正文
   - 基础清洗明显噪音内容

4. **结构化总结**
   - 调用 LLM 生成 JSON 结果
   - 对输出做基本校验
   - 保证输出结构稳定

5. **结果输出**
   - 在终端打印清晰结果
   - 支持输出 JSON
   - 可选保存运行产物

6. **基础文档**
   - README 说明配置、运行方式、输出结构
   - 提供最小使用示例

---

## 3. 本次明确不做

以下内容本次不要实现：

- Web UI
- 浏览器插件
- 用户系统
- 登录鉴权
- 历史记录管理
- 批量 URL 处理
- 多模型切换界面
- RAG
- Agent 化自动规划
- 登录态网页抓取
- 针对所有站点的兼容适配
- 复杂部署方案

重点是保证最小链路清楚，不要为了“看起来完整”而扩展范围。

---

## 4. 输入与输出要求

### 输入

- 必填：网页 URL
- 可选：输出语言
- 可选：是否保存运行结果
- 可选：保存目录

### 输出

模型输出必须符合如下 JSON 结构：

```json
{
  "title": "",
  "summary": "",
  "keyPoints": [],
  "actionItems": [],
  "tags": []
}
```

### 字段要求

- `title`：文章标题，允许为空但应尽量填充
- `summary`：整体摘要，简洁可读
- `keyPoints`：关键要点列表，建议 3~5 条
- `actionItems`：如内容中有明确行动建议则提取，否则可为空
- `tags`：主题标签，建议 2~5 个

---

## 5. 技术约束

### 技术栈

- Python 3.12+
- 使用 `uv` 管理依赖与运行
- 项目结构尽量简单清晰

### 依赖策略

允许使用少量成熟依赖来完成以下工作：
- HTTP 请求
- HTML 解析
- 正文提取
- 环境变量管理
- 数据校验
- LLM API 调用

不要为了“工程感”引入过多框架。

### 配置要求

以下内容从环境变量或 `.env` 读取：
- API key
- base URL
- model
- timeout
- output dir
- max input chars

不得硬编码敏感信息。

---

## 6. 质量要求

### 代码结构

- 不要把全部逻辑塞进一个文件
- 至少按职责拆分：配置、抓取、提取、总结、pipeline、CLI
- 命名要直观，不做过度抽象

### 错误处理

至少覆盖：
- 非法 URL
- 请求失败
- 请求超时
- 页面正文提取为空
- 模型输出不合法

错误信息应该可读，便于定位问题。

### 可维护性

- 主链路清晰
- 模块边界清楚
- README 能帮助他人快速运行
- 适度保留调试信息

---

## 7. 安全要求

- 不得硬编码密钥、token、账号信息
- 不得关闭 SSL 校验来“临时解决问题”
- 不得使用危险 shell 拼接
- 所有网络请求必须有超时
- 对输入 URL 做基本校验
- 对外部模型输出做解析与校验，不直接信任原始文本

---

## 8. 交付要求

完成后，交付物至少包括：

1. 可运行的项目代码
2. README
3. `.env.example`
4. 最小运行示例
5. 输出结构说明
6. 已知限制说明
7. 最小验证方式（可为样例运行或测试）

---

## 9. 已知限制应主动写明

请在交付说明中明确写出以下类型的限制：

- 某些依赖前端渲染的网页可能提取失败
- 某些页面可能只能抓到 loading 或框架内容
- 当前版本对超长文本只做截断，不保证完整覆盖
- 当前版本更适合静态内容较完整的网页

不要把“暂不支持”伪装成“已经支持”。

---

## 10. 编码前必须先输出的计划

在真正开始编码前，请先输出：

1. 计划新增或修改哪些文件
2. 每个文件承担什么职责
3. 主链路如何串起来
4. 主要风险点是什么
5. 你准备如何验证结果

未经计划说明，不要直接开始大段生成代码。

---

## 11. 交付前必须输出的自检

完成后请给出一份简短自检，至少包括：

- 本次实现了什么
- 没有实现什么
- 已知限制是什么
- 风险点是什么
- 如何运行与验证

---

## 12. 本文档的用途

这份文档适用于：

- 把项目重新交给 AI coding tool 生成第一版代码
- 复盘第一次 AI 编码输出的问题
- 对比“简单项目描述”和“结构化任务文档”在交付质量上的差异

如果后续实践证明某些部分过重或无效，再继续删减和收敛。
