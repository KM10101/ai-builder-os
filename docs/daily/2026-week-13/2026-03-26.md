## 实践 / 学习记录

### WSL 安装与目录规划

- 首次安装 WSL，未使用默认发行版目录，而是将 Ubuntu 安装到 `E:\wsls` 下。
- 明确了 WSL 组件本身仍属于 Windows 系统组件，能指定路径的是 Linux 发行版数据目录。

### 代理与网络

- 安装阶段遇到下载问题，先通过宿主机代理完成 WSL / 发行版安装。
- 后续为了让 WSL 内部访问 GitHub、npm 等站点更稳定，切换为可在 WSL 内使用宿主机代理的配置方式。
- 理解了 `localhost` 代理在 WSL NAT 模式下的限制，并完成可用方案调整。

### GUI / WSLg

- 初始状态下，WSL GUI 应用只能在任务栏预览，窗口无法正常激活。
- 通过重新启动 WSL 图形会话后恢复正常。
- 当前需求不需要完整桌面环境，只需要 WSLg 支持单个 GUI 应用即可。
- 后续补齐了中文语言支持，解决 GUI 应用中文乱码问题。

### 开发环境与工具

- 安装了常用基础包，包括 `git`、`curl`、`ripgrep`、`build-essential`、Python 相关包等。
- 计划在 WSL 内使用 Linux 侧的 Node.js / npm，避免与 Windows 环境混用。
- 明确后续 `Claude Code`、`Codex`、`Zed` 等开发工具优先在 WSL 环境中运行。

### 宿主机隔离与连接方式

- 为降低 AI 工具误操作宿主机文件的风险，关闭了 Windows 盘符在 WSL 中的自动挂载。
- 同时关闭了默认的 Windows 互操作能力，减少 WSL 对宿主机路径与命令的直接访问。
- 这样做会破坏 MobaXterm 的原生 WSL 集成方式，因此改为通过 SSH 连接 WSL。
- 结论上，WSL 可以做到“默认不暴露宿主机目录”，但如果追求更强的硬隔离，仍应使用完整虚拟机。
