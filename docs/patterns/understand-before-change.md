# 先理解，后改动

## 适用场景

适用于：
- 接手陌生模块
- 需要修改现有逻辑
- 不确定影响范围
- 需要减少误改概率

## 核心思想

在让 Claude Code / Codex 修改代码之前，先让它解释当前实现、入口、调用链和风险点。

这样做可以减少：
- 错误理解需求
- 对现有逻辑的误判
- 过早修改导致的返工

## 推荐步骤

1. 先要求它不要改代码
2. 让它找入口和关键文件
3. 让它解释现状
4. 让它指出风险点
5. 再决定是否进入修改阶段

## 常见起手语句

- 先不要改代码，先帮我理解这部分实现
- 先梳理入口、调用链和关键文件
- 先总结现状，再给修改建议

## 常见坑

- 一上来就让工具直接实现
- 没有限定分析范围
- 没要求它指出风险和依赖
