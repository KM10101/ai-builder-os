# LLM 应用基础概念笔记

## 这份笔记的目标

这份笔记用于建立对 LLM 应用的第一层整体理解。

目标不是一次性学全，而是先回答几个关键问题：

- LLM 应用到底是什么
- 一个基础应用通常由哪些部分组成
- 为什么需要 structured output、tool calling、RAG 这些能力
- workflow 和 agent 分别适合什么场景

这份笔记适合作为第一周的入门阅读材料。

## 1. 什么是 LLM 应用

LLM 应用不是简单地“调用一下大模型接口”，而是把大模型能力接入一个完整应用系统的过程。

可以先这样理解：

> LLM 负责生成和理解，应用层负责组织、约束、校验和集成。

一个最基础的 LLM 应用，通常会包含这些部分：

- 用户输入
- Prompt 构造
- 模型调用
- 输出解析
- 应用逻辑处理
- 返回结果
- 日志、错误处理、成本控制

所以，LLM 应用的重点从来不只是模型本身，还包括：

- 输入如何设计
- 输出如何约束
- 外部信息如何接入
- 应用逻辑如何组织
- 工程质量如何保证

## 2. Prompt

Prompt 是 LLM 应用里最基础的一层。

它不是一句随便写的话，而是对模型任务的定义方式。一个好的 prompt 往往会同时描述：

- 目标是什么
- 输入是什么
- 约束有哪些
- 输出应该长什么样

常见组成包括：

- system prompt
- user prompt
- context
- constraints
- output format

### 核心理解

Prompt 的本质不是“技巧”，而是**清晰表达任务和边界**。

如果 prompt 不清楚，模型就容易：

- 理解偏题
- 输出不稳定
- 漏掉关键约束
- 结果难以复用

### 学习重点

在学习 prompt 时，应该重点关注：

- 如何写清楚目标
- 如何写清楚约束
- 如何降低模型跑偏概率
- 如何让输出更稳定

## 3. Structured Output

Structured output 指的是：

> 不只是让模型返回一段自然语言，而是让它返回可以被程序稳定解析和继续使用的结构化结果。

常见形式包括：

- JSON
- 固定字段对象
- 枚举值
- 符合 schema 的输出

### 为什么它重要

如果只是聊天，自然语言结果通常已经够用。

但在应用场景里，结果往往还要继续被系统处理，例如：

- 存库
- 渲染页面
- 驱动后续逻辑
- 传给其他服务

如果输出不稳定，整个应用就会变得脆弱。

### 核心理解

structured output 的意义在于：

- 让结果更可控
- 让结果更可解析
- 让结果更容易验证
- 让 LLM 从“会说”变成“能接入系统”

## 4. Tool Calling / Function Calling

Tool calling 或 function calling 是 LLM 应用中非常关键的一层。

它的本质不是模型真的自己执行工具，而是：

> 模型根据当前任务，判断应该调用哪个工具，并生成对应参数；真正的工具执行由应用程序完成。

例如，模型可以提出这些调用意图：

- 查询天气
- 搜索网页
- 调数据库
- 调内部 API
- 发送消息

### 正确理解方式

需要分清两层：

- 模型层：决定“该调什么工具、传什么参数”
- 应用层：真正执行工具，并把结果返回给模型或业务逻辑

### 为什么重要

没有 tool calling 时，模型更多是在“说”。

有了 tool calling 之后，模型才可能和外部世界建立真实连接，进入“能做事”的状态。

这是后续做 workflow、agent、多工具系统的重要基础。

## 5. Context Management

Context management 指的是：

> 这次模型调用到底要看到哪些信息，这些信息怎么组织，放多少，按什么顺序放。

这件事非常重要，因为上下文并不是越多越好。

### 它通常涉及的问题

- 要不要带历史对话
- 要不要带外部资料
- 带多少资料合适
- 多个信息块怎么排序
- 如何避免无关信息污染结果

### 核心理解

上下文既影响：

- 输出质量
- 成本
- 延迟
- 稳定性

所以 context management 本质上是一种质量控制手段，而不只是“拼接文本”。

## 6. RAG

RAG 是 Retrieval-Augmented Generation 的缩写，通常翻译为“检索增强生成”。

它的核心思想是：

> 不让模型只依赖训练时记住的内容，而是先检索相关资料，再基于资料生成答案。

这类应用非常常见，例如：

- 知识库问答
- 文档助手
- 企业内部问答
- FAQ 助手

### 基础链路

一个基础 RAG 应用通常包含：

- 文档切分
- 向量化（embedding）
- 检索（retrieval）
- 选取相关片段
- 拼接上下文
- 让模型回答

### 核心理解

RAG 解决的问题不是“让模型更聪明”，而是：

- 让模型基于指定资料回答
- 降低胡说概率
- 提高回答的相关性和可控性

## 7. Workflow 与 Agent

这两个概念很容易混在一起，但最好尽早区分清楚。

### Workflow

workflow 通常指：

- 步骤固定
- 流程明确
- 顺序可控
- 更容易测试和调试

例如：

1. 获取网页内容
2. 清洗正文
3. 调模型做总结
4. 输出结构化结果

### Agent

agent 通常指：

- 模型参与决策
- 路径不完全固定
- 可以根据中间结果决定下一步动作
- 更灵活，但也更难控制

例如：

1. 先搜索信息
2. 根据搜索结果决定是否继续搜索
3. 再决定是否调用其他工具
4. 最后汇总输出

### 初学阶段怎么选

大多数第一阶段项目，更适合先做 workflow，而不是直接做 agent。

原因是 workflow：

- 更容易控
- 更容易验证
- 更容易 debug
- 更适合作为应用第一版

## 8. Evaluation

Evaluation 指的是：

> 如何判断一个 LLM 应用的输出是不是好用、稳定、可信、值得继续迭代。

很多人做 LLM 应用时，容易只凭感觉判断效果，但这通常不够。

### 初学阶段至少应该关注

- 输出格式是否稳定
- 内容是否满足要求
- 是否容易胡说
- 结果是否可复现
- 速度和 token 成本是否可接受

### 核心理解

evaluation 不是额外工作，而是保证应用质量的必要部分。

如果没有评估，应用就很难稳定改进。

## 9. 一个最小 LLM 应用链路应该怎么理解

可以先用下面这个最小链路来理解整个系统：

1. 用户输入问题或任务
2. 应用构造 prompt 和上下文
3. 调用模型
4. 拿到结构化结果
5. 应用对结果做解析和处理
6. 返回最终输出

如果引入工具或检索，链路会继续扩展，但核心思路不变：

- 模型负责生成和决策的一部分
- 应用负责组织、执行、控制和校验

## 10. 现阶段最值得先记住的几个结论

### 结论 1

LLM 应用不是单纯调 API，而是“模型能力 + 应用系统”的结合。

### 结论 2

Prompt 的核心不是花哨，而是任务表达清晰、约束明确。

### 结论 3

structured output 是从 demo 走向应用的关键一步。

### 结论 4

tool calling 让模型从“会说”逐渐变成“能协助做事”。

### 结论 5

RAG 的重点不是模型变聪明，而是让回答建立在指定资料上。

### 结论 6

大多数第一阶段项目，先做 workflow 比直接做 agent 更稳。

### 结论 7

evaluation 不是最后才做，而是从一开始就应该有意识。

## 11. 明天阅读时可以重点思考的问题

明天学习时，可以边读边试着回答下面这些问题：

- 一个 LLM 应用和普通后端服务最大的差异是什么？
- 为什么自然语言输出在应用里经常不够用？
- tool calling 和普通 API 调用的关系是什么？
- context 为什么不是越多越好？
- workflow 和 agent 哪种更适合自己的第一阶段项目？
- 如果做一个网页内容总结工具，里面会用到哪些概念？
